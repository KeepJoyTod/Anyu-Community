**P1**

这部分决定项目是否能稳定维护，而不是上线后迅速失控。

- `接口契约标准化`
  - 废弃 README 中“无 DTO 模式”的开发约定，见 [README.md](</d:/Java/projectku/安隅智能社区/README.md:62>)。
  - 所有核心接口从 `Map<String, Object>` 改为 `Request DTO + Response VO + @Valid`。
  - 先改居民端高频接口：登录、注册、报修、投诉、公告、场地预约、积分、社区互动。
  - 统一错误码、分页结构、时间字段格式、空值策略。

- `重复代码清理`
  - 清理 `*1.java`、`*1.vue`、`*1.xml`、`.bak` 文件，当前这类文件约 `93` 个。
  - 明确保留版本，删除废弃分支，避免运行时误扫包、误引用、维护分叉。
  - 重点目录：`ruoyi-framework`、`ruoyi-system`、`ruoyi-home`、`ruoyi-ui`。

- `模块边界重构`
  - 当前后端 `17` 个模块，业务拆分过细但边界不清。
  - 建议重组为：`common`、`system/auth`、`property-service`、`community-service`、`notification`、`admin-api`。
  - 统一命名，修正 `visiter` 这类拼写问题，避免长期污染接口和表结构。

- `数据库治理`
  - 把 SQL 脚本初始化改为 Flyway 或 Liquibase。
  - 建表规范统一：主键、唯一索引、逻辑删除、审计字段、状态枚举、外键策略。
  - 对缴费、积分、预约、投诉、报修补唯一约束和状态流转约束。
  - 检查事务边界和幂等性，尤其是支付、积分发放、报名取消。

- `日志与审计`
  - 操作日志、登录日志、业务日志分层。
  - 敏感信息脱敏，禁止打印 token、手机号、身份证类信息。

**P2**

这部分决定“像不像企业产品”。

- `测试体系`
  - 后端补单元测试、Service 集成测试、Controller 接口测试。
  - 前端补 API 层测试和关键流程 E2E。
  - 当前我没有发现有效测试用例，这必须改变。

- `性能与稳定性`
  - 建压测脚本，覆盖登录、首页、公告、社区列表、报修提交。
  - 增加 Redis 缓存策略、热点接口缓存、分页优化、SQL 慢查询治理。
  - 文件资源接入 CDN/对象存储，避免本地磁盘成为瓶颈。

- `可观测性`
  - 接入健康检查、指标采集、异常告警、链路追踪。
  - 至少具备 JVM、线程池、DB 连接池、Redis、接口耗时、错误率监控。
  - 关键业务告警：登录失败突增、上传失败、支付失败、AI 调用失败。

- `前端工程化`
  - API 封装统一，删除 `request.js/request1.js/index1.js` 这类并存实现。
  - 页面状态、权限跳转、异常提示、空态、重试机制统一。
  - 静态大图压缩，当前资源体积偏大，移动端首屏会受影响。

- `文档与交付`
  - 补齐架构图、模块图、ER 图、接口规范、部署手册、回滚手册、值班手册。
